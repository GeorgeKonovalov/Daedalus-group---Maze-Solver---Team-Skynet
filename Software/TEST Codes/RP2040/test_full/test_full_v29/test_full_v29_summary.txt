New IMU work in this version
----------------------------

- Added tolerance-aware yaw-hold compensation using the onboard LSM6DSOX gyroscope
- Yaw hold is relative to the saved start frame; the board has no magnetometer, so this is not an absolute compass heading
- Added `IMU Test` to the root menu for raw `ax ay az gx gy gz` visibility
- Added `Yaw Test` to the root menu for a yaw-only validation view
- Added `IMU PID Test` to the PID menu with:

  - propagated deadband based on official sensor tolerances
  - live rotational compensation preview
  - short-press capture of the current body frame as the new initial coordinates
- Added editable `IMU P`, `IMU I`, `IMU D`, and `IMU threshold` settings in the PID menu
- Renewed the X-drive mixer to combine translation and rotational compensation without renormalizing away the commanded left/right direction

LSM6DSOX tolerance model used
-----------------------------
- Gyroscope zero-rate level: `+-1.0 dps`
- Gyroscope RMS noise: `0.075 dps`
- Gyroscope sensitivity tolerance: `+-1%`
- Accelerometer zero-g level offset: `+-20 mg`
- Accelerometer RMS noise: `1.8 mg`

These values are used to build the IMU deadband as:
- reference uncertainty from the capture window
- planar accelerometer uncertainty from the saved initial coordinates
- live measurement uncertainty from current yaw error and gyro rate tolerance
- user-tunable extra angular threshold on top

Architecture example
--------------------
1. `refreshSensors()` updates IR, ultrasonic and IMU state each loop.
2. `beginExecution()` or `startEventTestRun()` calls `captureImuReference()` to save:
   - the current body x-axis vector in the robot plane
   - the averaged initial `ax, ay, az`
   - the propagated angular tolerance of that saved frame
3. `followSmart()` computes:
   - wall PID lateral correction
   - IMU rotational correction from the signed angle between `x_initialised` and the live x-axis vector
4. `driveRobotFrame(lateral, forward, rotation, heading)` sends those commands into the 45 degree X-drive mixer:
   - `u+ = vy + vx` for the FL/RR diagonal
   - `u- = vy - vx` for the FR/RL diagonal
   - `spin = [+w, -w, -w, +w]` for yaw compensation
5. `w` is clamped only to the remaining wheel headroom, so pure left/right translation keeps its original translational pattern mathematically instead of being distorted by post-mix normalization.

Yaw-only test workflow
----------------------
1. Open `Yaw Test` from the root menu.
2. Keep the robot still and short-press once to save the current heading as zero.
3. Rotate the robot around the vertical axis.
4. Read the displayed relative yaw angle in degrees.

The shown yaw is relative to the saved reference heading, so it is the right test for checking rotational compensation behavior.
